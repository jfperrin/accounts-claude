# 2FA — Email + Application d'authentification (TOTP)

## Contexte

L'application gère actuellement l'authentification par mot de passe (Passport local)
et par OAuth Google. Aucun second facteur n'est requis. On ajoute le support du 2FA :
- par **application d'authentification** (TOTP, RFC 6238)
- par **code email** à usage unique
Les deux peuvent être activés simultanément sur un même compte.

## Décisions de design

| Décision | Choix retenu |
|----------|--------------|
| Activation | Optionnelle, par utilisateur depuis son profil |
| Méthodes simultanées | TOTP + email peuvent être actifs en même temps |
| Recovery codes | 10 codes alphanumériques de 10 caractères (ex. `a4b2c9d1e3`), affichés une seule fois, hashés bcrypt |
| Comptes Google OAuth | Pas de 2FA local (Google gère le sien) |
| Flux login (les deux actifs) | Page intermédiaire `/login/mfa` avec choix `Authenticator` / `Email` |
| Setup email | Code de test obligatoire avant activation |
| Désactivation | Mot de passe + code 2FA actuel exigés |
| Format code email | 6 chiffres, valide 10 minutes |
| Stockage `totpSecret` | Chiffré AES-256-GCM en DB |

## Modèle de données

### Champs ajoutés sur `User`

| Champ | Type | Sémantique |
|-------|------|------------|
| `totpSecret` | `string \| null` | Secret base32 chiffré (`v1:iv:ciphertext:tag`). `null` si TOTP inactif. |
| `totpEnabled` | `boolean` | `false` tant que la config n'a pas été confirmée par un code valide |
| `emailMfaEnabled` | `boolean` | `true` si le code email est un facteur valide au login |
| `recoveryCodes` | `string[]` | Tableau de hashs bcrypt. Un code consommé est retiré du tableau. |

### Nouvelle table `mfa_email_codes` / collection `MfaEmailCode`

| Colonne | Type | Notes |
|---------|------|-------|
| `id` / `_id` | UUID / ObjectId | PK |
| `userId` | FK User | NOT NULL |
| `codeHash` | string | bcrypt du code 6 chiffres |
| `purpose` | enum | `'login' \| 'setup' \| 'disable'` |
| `expiresAt` | datetime | `createdAt + 10min` |
| `used` | boolean | `true` à la 1re vérification (succès ou échec) — un code = un seul essai |
| `createdAt` | datetime | |

Cleanup au boot du serveur : `DELETE WHERE expiresAt < now() - 1 day`.

### Backport SQLite

Dans `server/db/sqlite.js`, bloc try/catch idempotent :
```sql
ALTER TABLE users ADD COLUMN totp_secret TEXT;
ALTER TABLE users ADD COLUMN totp_enabled INTEGER NOT NULL DEFAULT 0;
ALTER TABLE users ADD COLUMN email_mfa_enabled INTEGER NOT NULL DEFAULT 0;
ALTER TABLE users ADD COLUMN recovery_codes TEXT; -- JSON string

CREATE TABLE IF NOT EXISTS mfa_email_codes (
  id          TEXT PRIMARY KEY,
  user_id     TEXT NOT NULL REFERENCES users(id),
  code_hash   TEXT NOT NULL,
  purpose     TEXT NOT NULL,
  expires_at  TEXT NOT NULL,
  used        INTEGER NOT NULL DEFAULT 0,
  created_at  TEXT DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_mfa_codes_user ON mfa_email_codes(user_id);
```

`mapUser` exposera `totpEnabled`, `emailMfaEnabled`, et `recoveryCodes` parsé depuis le JSON. `mapMfaCode` traduit `code_hash`, `expires_at`, `used`.

- Ajout des 4 champs sur `models/User.js` (`totpSecret String`, `totpEnabled Boolean default:false`, `emailMfaEnabled Boolean default:false`, `recoveryCodes [String] default:[]`).
- Nouveau `models/MfaEmailCode.js` avec les mêmes champs.

## API serveur

Toutes les routes ci-dessous sont préfixées `/api/auth/mfa` et montées depuis `routes/auth.js`. Le router lui-même vit dans `routes/mfa.js`.

### Gestion (utilisateur connecté)

| Route | Body | Effet |
|-------|------|-------|
| `POST /totp/setup` | — | Génère un secret base32, le stocke (chiffré) avec `totpEnabled=false`. Renvoie `{ secret, otpauthUrl, qrCodeDataUrl }`. |
| `POST /totp/enable` | `{ code }` | Vérifie le code TOTP. Si valide → `totpEnabled=true`, génère 10 recovery codes, renvoie `{ recoveryCodes: [...] }` (une seule fois, en clair). |
| `POST /totp/disable` | `{ password, code }` | Vérifie password + code (TOTP ou recovery ou email si actif). Si OK → efface `totpSecret`, `totpEnabled=false`. Si plus aucun facteur actif, vide `recoveryCodes`. |
| `POST /email/setup` | — | Génère un code 6 chiffres, envoie par email, stocke hashé avec `purpose='setup'`. Rate-limit 1/60s. |
| `POST /email/enable` | `{ code }` | Vérifie le code. Si OK → `emailMfaEnabled=true`. Si TOTP non actif, génère aussi les recovery codes. |
| `POST /email/disable` | `{ password, code }` | Idem TOTP. |
| `POST /recovery/regenerate` | `{ password, code }` | Invalide les anciens, génère 10 nouveaux, renvoie en clair une seule fois. |

Tous les endpoints `/mfa/*` rejettent un compte Google (`user.googleId && !user.passwordHash`) avec `400 { message: '2FA non disponible pour les comptes Google' }`.

### Login modifié

`POST /api/auth/login` change de contrat :
- Si l'utilisateur n'a aucun MFA actif (`!totpEnabled && !emailMfaEnabled`) ou est un compte Google → comportement actuel inchangé (réponse `200 { user }`, session active).
- Sinon : ne pas appeler `req.login()`. Stocker en session :
  ```js
  req.session.mfaChallenge = {
    userId,
    rememberDays,
    methods: ['totp', 'email'].filter(...),  // celles actives
    createdAt: Date.now(),
  };
  ```
  Réponse : `200 { mfaRequired: true, methods: ['totp', 'email'] }`.

### Challenge (session avec `mfaChallenge`, pas encore `isAuthenticated()`)

| Route | Body | Effet |
|-------|------|-------|
| `POST /mfa/challenge/send-email` | — | Requiert `email` dans `methods`. Génère/envoie un code, stocke hashé avec `purpose='login'`. Rate-limit 1/60s, 3/15min. |
| `POST /mfa/challenge/verify` | `{ method, code }` (`method ∈ totp \| email \| recovery`) | Vérifie. Si OK : `req.login(userId)`, applique `rememberDays`, efface `mfaChallenge`, renvoie le user sérialisé. |
| `POST /mfa/challenge/cancel` | — | Efface `mfaChallenge`. |

Validité du `mfaChallenge` : 10 min (vérifié à chaque appel via `createdAt + 600_000 < now`). Au-delà → 401 "Challenge expiré", invalidation et retour à `/login`.

Compteur d'échecs : `req.session.mfaChallenge.failedAttempts`. À 5, on invalide le challenge.

`GET /api/auth/me` étendu : ajoute `totpEnabled`, `emailMfaEnabled`, `recoveryCodesRemaining` (longueur du tableau).

## API client (`client/src/api/mfa.js`)

```js
export const setupTotp        = () => client.post('/api/auth/mfa/totp/setup');
export const enableTotp       = (code) => client.post('/api/auth/mfa/totp/enable', { code });
export const disableTotp      = (password, code) => client.post('/api/auth/mfa/totp/disable', { password, code });
export const setupEmailMfa    = () => client.post('/api/auth/mfa/email/setup');
export const enableEmailMfa   = (code) => client.post('/api/auth/mfa/email/enable', { code });
export const disableEmailMfa  = (password, code) => client.post('/api/auth/mfa/email/disable', { password, code });
export const regenerateRecovery = (password, code) => client.post('/api/auth/mfa/recovery/regenerate', { password, code });
export const sendChallengeEmail = () => client.post('/api/auth/mfa/challenge/send-email');
export const verifyChallenge    = (method, code) => client.post('/api/auth/mfa/challenge/verify', { method, code });
export const cancelChallenge    = () => client.post('/api/auth/mfa/challenge/cancel');
```

- `LoginPage.jsx` : si `login()` renvoie `{ mfaRequired, methods }`, on `navigate('/login/mfa')`.
- **Nouvelle page** `pages/MfaChallengePage.jsx`, montée sur `/login/mfa` (publique). Comportement :
  - Si pas de challenge en mémoire (reload) → redirige `/login`.
  - `methods.length === 1` : saute l'écran de choix, affiche directement le champ pour cette méthode (envoie automatiquement le code email si c'est la seule méthode).

  - Champ code + bouton Valider, lien "Utiliser un code de récupération" (toujours visible), lien "Annuler" → `/login`.

  - `login()` : si la réponse est `{ mfaRequired }`, met à jour `mfaChallenge` et renvoie cette valeur ; sinon comportement actuel.
  - Nouvelles méthodes : `verifyMfa(method, code)`, `cancelMfa()`, `sendMfaEmail()`.
- **`App.jsx`** : route publique `/login/mfa` ajoutée à côté de `/login`.

## Sécurité

### Rate limits

| Endpoint | Limite |
|----------|--------|
| `POST /mfa/challenge/verify` | 5 tentatives / 15min par IP, **et** 5 par challenge (compteur session) |
| `POST /mfa/challenge/send-email` | 1 / 60s, 3 / 15min par user |
| `POST /mfa/email/setup` | 1 / 60s, 3 / 15min par user |
| `POST /mfa/totp/setup` | 5 / 15min par user |

### Anti-énumération

`POST /mfa/challenge/verify` répond `401 { message: 'Code invalide' }` indifféremment pour code TOTP faux, email expiré, recovery déjà utilisé, challenge expiré.

### Comparaison constant-time

- TOTP : `otplib` est constant-time par défaut.
- Recovery & codes email : `bcrypt.compare`.

### Chiffrement `totpSecret`

Nouveau fichier `server/utils/cryptoBox.js` :

```js
// AES-256-GCM, clé hex 32 bytes dans MFA_ENCRYPTION_KEY.
// Format de sortie : "v1:<iv-hex>:<ciphertext-hex>:<tag-hex>"
exports.encrypt = (plain) => { ... };
exports.decrypt = (token) => { ... };
```

Variable `MFA_ENCRYPTION_KEY` (64 hex chars = 32 bytes). Au boot, si la var est absente :
- log un warning "MFA_ENCRYPTION_KEY non défini — les comptes avec TOTP seront cassés"
- le serveur démarre quand même (éviter de bloquer dev sans MFA configuré)

### TTL et cleanup

- `mfa_email_codes.expiresAt = now + 10min`.
- `used=true` à la 1re vérification réussie ou échouée → 1 essai par code email.
- Cleanup au boot : `DELETE FROM mfa_email_codes WHERE expiresAt < now - 1day`.
- `mfaChallenge` en session : TTL 10 min vérifié à chaque requête.

### Sessions actives

Activer ou désactiver un facteur ne touche **pas** aux sessions existantes (cohérent avec le change de password actuel). Pas de logout forcé.

## Tests

`server/tests/mfa.test.js` couvre :

- Setup TOTP : code de confirmation valide active, code invalide rejette en 401.
- Login TOTP complet : login → `mfaRequired`, verify → user authentifié.
- Login email : `send-email` puis verify → succès.
- Recovery code : verify avec recovery → succès ; rejoué → 401.
- 5 tentatives invalides → challenge invalidé.
- Cross-user : challenge user A, code user B → 401.
- Disable sans password → 400 ; sans code 2FA → 400 ; complet → désactivation.
- Compte Google → routes `/mfa/*` rejetées en 400.
- Cleanup : `mfa_email_codes` expirés purgés au boot.

Pas de test client ajouté (le projet n'a qu'un seul fichier de test client aujourd'hui).
